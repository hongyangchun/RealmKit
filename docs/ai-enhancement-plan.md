# RealmKit AI 增强方案分析

> 分析日期：2026-05-17  
> 目标：梳理现有 AI 功能，规划世界构建过程中 AI 的深度融入方案

---

## 一、现有 AI 功能盘点

### 1.1 AI 服务架构

项目通过 `src/services/aiService.ts` 提供统一的 LLM 调用层，采用 OpenAI Chat Completions 兼容格式，支持多种 LLM 提供商。

| 能力 | 说明 |
|------|------|
| 调用方式 | 原生 `fetch()` POST，OpenAI 兼容格式 |
| 配置存储 | `localStorage` 键 `zzworld_ai_config` |
| 超时控制 | 60 秒 `AbortController` |
| 错误处理 | 401/403 → 密钥无效，429 → 频率限制 |
| 预设提供商 | GPT-4o / GPT-4o Mini / Claude 3.5 / DeepSeek V3 / Qwen 2.5 / 自定义 |

### 1.2 已实现的 AI 功能（2 项）

| 功能 | 入口位置 | AI 必需 | 降级方案 |
|------|---------|---------|---------|
| **编年史生成** | ChroniclePage → ChronicleForm | 是（无降级） | 弹出 AI 配置引导 |
| **角色剧情建议** | CharacterDetailPanel → PlotSuggestionPanel | 否 | 17 个特质模板自动匹配 |

#### 功能 A：编年史生成

- **触发**：用户选择年份范围和标题 → 点击"撰写新章节"
- **上下文构建**：`_buildWorldContext()` 汇总势力、人物（含特质/传记/生卒年）、时间段内事件
- **Prompt 设计**：中文系统提示，要求文学性叙事，500-800 字，适合 12 岁阅读
- **存储**：独立的 `ChronicleEntry`，存于 `localStorage` 键 `zzworld_chronicles`

#### 功能 B：角色剧情建议

- **双模式架构**：
  - AI 模式：发送人物完整信息（特质/技能/传记），要求返回 JSON 格式的 3 条剧情
  - 模板模式：17 个中文特质 → 34 个预设剧情模板，随机匹配 + 占位符替换
- **交互**：AI 失败自动降级到模板，用户无感知
- **一键创建**：建议可直接转为时间轴事件

### 1.3 程序化生成功能（非 AI，5 项）

| 功能 | 文件 | 技术 |
|------|------|------|
| 世界种子生成 | `worldSeedGenerator.ts` | 4 种风格名称库 + 模板引擎 |
| 地图地形生成 | `mapGenerator.ts` | Simplex Noise 噪声算法 |
| 头像生成 | `AvatarGenerator.ts` | Hash → 5x5 对称几何图案 |
| 故事种子 | `storySeedService.ts` | 冲突 → 叙事谜题模板 |
| 关系叙事 | `graphNarrativeService.ts` | 关系类型 → 文本模板 |

## 二、AI 融入世界创建的 6 个方向

### 方向 A：AI 世界种子增强（P0 核心体验）

**现状**：世界种子通过名称库模板随机组合，风格固定、缺乏个性化。

**增强方案**：

```
用户输入主题关键词（如"冰与火之歌""东方仙侠""蒸汽朋克"）
    ↓
LLM 生成完整世界观框架：
    - 世界名称 + 简介描述
    - 势力名称/特色/描述（3-6 个）
    - 代表性人物名字/特质/背景（每势力 2-3 人）
    - 地理特色（地形风格、命名风格）
    - 文化元素（语言风格、宗教/信仰、技术等级）
    ↓
结构化 JSON 返回 → 自动填入 WorldSeedResult
    ↓
用户预览 → 调整 → 确认创建
```

**技术实现**：
- 新增 `src/services/aiWorldSeedService.ts`
- Prompt 要求 LLM 返回固定 JSON schema，匹配现有 `WorldSeedResult` 类型
- 保留现有模板引擎作为离线降级
- WorldSeedWizard 第一步新增"AI 模式"开关

**Prompt 设计要点**：
```
System: 你是一位架空世界观设计师。根据用户提供的主题关键词，
设计一个完整的世界观框架。以 JSON 格式返回。
字段包括：worldName, description, factions[], characters[], 
terrainStyle, cultureNotes。

要求：
1. 势力之间要有矛盾冲突点
2. 人物要有鲜明的性格特质
3. 语言适合12岁阅读，生动有画面感
4. 保持内在逻辑一致性
```

---

### 方向 B：AI 地理命名与描述（P2 锦上添花）

**现状**：城市/地标名称来自预设名称库，风格固定。

**增强方案**：
- 地图生成后，调用 LLM 为每个城市/区域生成：
  - 特色名称（匹配世界风格）
  - 文化描述（50-100 字）
  - 地方传说（1-2 句）
- 批量调用，一次性为整个地图生成所有描述

**技术实现**：
- 新增 `src/services/aiGeoNamingService.ts`
- 将地图网格数据（地形分布 + 势力分布）压缩为文本描述传给 LLM
- 返回 JSON 格式：`{ locations: [{ id, name, description, legend }] }`

---

### 方向 C：AI 角色深化引擎（P0 核心体验）

**现状**：创建角色时需手动填写所有字段，新手门槛高。

**增强方案**：

```
用户输入：名字 + 所属势力 + 一句话定位（如"忠诚的骑士""神秘的术士"）
    ↓
LLM 一键生成完整人物画像：
    - 特质（2-4 个，如"勇敢、固执、正义感强"）
    - 背景 biography（100-200 字）
    - 技能（2-3 个，含名称和类型）
    - 建议关系（与同势力/对立势力人物的潜在关系）
    ↓
用户预览 → 逐字段编辑 → 确认
```

**技术实现**：
- 新增 `src/services/aiCharacterService.ts`
- CharacterForm 新增"AI 一键生成"按钮
- 需传入当前世界上下文（势力列表 + 已有人物），确保新人物融入现有格局
- 模板降级：基于势力风格 + 特质库随机组合

**交互设计**：
- 表单顶部显示"AI 深化"开关（仅在 AI 已配置时激活）
- 点击后弹出预览弹窗，用户可逐项接受/修改/重新生成
- 生成期间显示 loading 动画（"正在构思角色命运..."）

---

### 方向 D：AI 事件推演引擎（P1 深度增强）

**现状**：事件完全由用户手动创建，缺乏因果逻辑引导。

**增强方案**：

```
当前世界状态（势力 + 人物 + 关系 + 已有事件）
    ↓
LLM 分析并推演：
    - 现有矛盾点（势力对立、人物关系紧张）
    - 可能爆发的事件（战役/阴谋/外交/探险）
    - 因果链（上一个事件 → 下一个合理发展）
    - 转折点（意料之外的命运变化）
    ↓
返回 3-5 个事件建议，每个包含：
    - 年份建议（基于时间轴间隙）
    - 事件标题和描述
    - 涉及势力/人物
    - 前因后果说明
```

**技术实现**：
- 新增 `src/services/aiEventPredictor.ts`
- 将时间轴事件序列化为文本，构建因果链上下文
- TimelinePage 新增"推演未来"按钮
- 事件建议可一键添加到时间轴

---

### 方向 E：AI 关系网络分析（P1 深度增强）

**现状**：关系网络图仅可视化，缺乏深度分析。

**增强方案**：

```
关系图数据（人物 + 关系 + 势力归属）
    ↓
LLM 分析势力格局：
    - 权力平衡报告（哪个势力最强/最弱，为什么）
    - 潜在联盟建议（基于人物关系网络）
    - 矛盾预警（关系紧张但尚未爆发冲突的人物对）
    - 孤立节点识别（与其他人物没有关系的重要角色）
    ↓
返回分析报告（200-300 字）+ 关系建议列表
```

**技术实现**：
- 新增 `src/services/aiRelationAnalyzer.ts`
- GraphPage / CharacterPage 网络图模式新增"AI 分析"按钮
- 分析结果以浮层/抽屉形式展示
- 关系建议可一键创建新关系

---

### 方向 F：AI 编年史增强（P2 锦上添花）

**现状**：编年史生成已有基础功能，但风格单一。

**增强方案**：

1. **多种叙事风格**：
   - 史诗风格（大气磅礴，像荷马史诗）
   - 民间故事（口语化，像说书人）
   - 学术编年（严谨客观，像历史教科书）
   - 童话风格（温暖有趣，像给孩子讲故事）

2. **多章节连贯性**：
   - 记录已生成章节的核心内容摘要
   - 新章节生成时传入前文摘要，确保叙事连贯
   - 避免重复提及相同细节

3. **角色视角叙事**：
   - 选择一个角色，从该角色视角重述事件
   - 第一人称叙事，更具沉浸感

**技术实现**：
- 增强 `aiService.ts` 的 `generateChronicle` 方法
- 新增 `ChronicleStyle` 类型
- 新增章节摘要存储（`chronicleSummaries` 字段）
- ChroniclePage 表单新增风格选择器 + 角色视角选择器

## 三、AI 功能开关与配置优化

### 3.1 独立功能开关

当前 AI 配置只有全局开关（配置了就全部启用）。建议每个功能独立控制：

```typescript
interface AiFeatureFlags {
  worldSeedEnhanced: boolean;   // AI 世界种子
  characterDeepDive: boolean;   // AI 角色深化
  eventPrediction: boolean;     // AI 事件推演
  relationAnalysis: boolean;    // AI 关系分析
  geoNaming: boolean;           // AI 地理命名
  chronicleEnhanced: boolean;   // AI 编年史增强
}
```

### 3.2 调用统计与成本控制

- 记录每次 AI 调用的 token 消耗（估算）
- 在设置页展示使用统计
- 可设定每日/每月调用上限

## 五、技术注意事项

### 5.1 现有 Claude 预设兼容性问题

当前 Claude 3.5 Sonnet 预设使用 `api.anthropic.com/v1/messages`，但 `aiService.callLlm()` 发送的是 OpenAI 格式。如果用户没有代理/适配器，这个预设无法正常工作。

**建议**：要么移除 Claude 直连预设，要么在 `callLlm` 中增加 Anthropic API 格式适配。

### 5.2 API Key 安全

当前 API Key 明文存储在 `localStorage`。虽然纯前端应用的选择有限，但建议：
- 在 UI 中增加更醒目的安全提醒
- 提供"仅本次会话"选项（sessionStorage）
- 长期考虑接入 Cloudflare Workers 代理

### 5.3 并发与性能

- 多个 AI 功能同时调用时需要并发控制
- 建议增加全局请求队列，避免同时发送过多请求
- 考虑请求去重（相同上下文短时间内不重复调用）

### 5.4 Prompt 版本管理

- 当前 Prompt 硬编码在 TypeScript 文件中
- 建议将 Prompt 模板提取为独立配置文件（如 JSON/YAML）
- 便于后续 A/B 测试和调优

## 六、用户体验设计原则

1. **AI 可选而非强制**：所有 AI 功能都有模板降级，不配置 AI 也能完整使用
2. **即时反馈**：AI 生成过程中显示生动的 loading 提示（如"正在构思角色命运..."）
3. **可预览可编辑**：AI 生成的内容先预览，用户可逐项修改再确认
4. **一键重新生成**：对不满意的结果可以快速重试
5. **上下文感知**：AI 生成时传入完整世界上下文，确保新内容融入现有格局
6. **适龄语言**：所有生成内容适合 12 岁阅读，语言生动有画面感

---

## 七、总结

RealmKit 目前有 **2 项 AI 功能**（编年史生成 + 角色剧情建议）和 **5 项程序化生成功能**。AI 融入程度较低，主要集中在"后期加工"阶段（编年史/剧情建议），而非"创建阶段"。

本方案提出 **6 个 AI 增强方向**，核心思路是将 AI 从"后期辅助"前移到"创建即 AI"——在世界种子生成、角色创建、事件构建等核心环节深度融入 AI，让用户只需输入简单关键词就能获得丰富、连贯、个性化的世界内容。同时保持所有功能的离线可用性，确保不配置 AI 的用户也能完整使用所有功能。
